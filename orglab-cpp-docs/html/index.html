Learning About orglab_cpp

Besides the Doxygen-generated documention, the best way to learn about orglab_cpp is to read through the examples thoroughly- almost all concepts are covered. Go in this order:

  1. orglab_cpp-examples
  2. linalg_example
  3. automation_example
  4. threaded_realtime_example

orglab-cpp Installation

orglab_cpp requires a local installation of Origin 2020b or later.

Simply put the orglab_cpp folder somewhere in your VC++ project source and use this include (changing path as necessary): #include "orglab_cpp/all.h". VC++ projects should only be compiled for x64.

By default OrgLab is used but the Origin COM Automation Server can be used by adding this #define prior to the above mentioned #include: #define ORGLAB_CPP_USE_ORIGIN. Otherwise OrgLab will be used.

Known Limitations

  • orglab-cpp is 64-bit only.
  • orglab-cpp No support for deleting objects but raw COM pointers may be accessed and used for this purpose via the class method raw_ptr().
  • orglab-cpp Graph and LabTalk functionality is limited to the Automation Server.
  • orglab-cpp Layout pages are not implemented.
  • OrgLab/Automation Server Limitation - Only one project can be worked on at a time.
  • OrgLab/Automation Server Limitation - No support for User-Defined label rows via pure COM methods.
  • OrgLab/Automation Server Limitation - When NaNs are returned in column and matrix data, the value isn't a C++ Nan, it is:
    #define OLCPP_ORIGIN_NANUM -1.23456789E-300
    However, to utility functions are available to convert from and to that value when reading or writing:
    /*! Convert from Origin nanum to C++ nan if necessary */
    template<class T>
    inline T check_from_origin_nan(const T& val) {
    if (OLCPP_ORIGIN_NANUM == val)
    return std::numeric_limits<T>::quiet_NaN();
    return val;
    }
    /*! Convert from numeric value from C++ nan to Origin nanum if necessary */
    template<class T>
    inline T check_to_origin_nan(const T& val) {
    if (std::numeric_limits<T>::quiet_NaN() == val)
    return OLCPP_ORIGIN_NANUM;
    return val;
    }

Important Information about OrgLab and Automation Server

  • Though the COM interface seems inheritance-based, it has gaps and cannot be translated to C++ class inheritance (there will be compiler errors that cannot be overcome). Hence, orglab_cpp classes are not inheritance-based and there is repetition of methods.
  • OrgLab is mainly for writing and reading data (workbooks, matrixbooks, sheets, columns, and matrices).
  • Automation Server supports writing and reading data as well as creating graphs and executing LabTalk commands.
  • OrgLab is very fast.
  • Automation is slower, especially at start up.
  • OrgLab is DLL-based and has no visual presence.
  • Automation server can be visible or hidden.
  • The graphing functionality in Automation Server is somehat limited. However, LabTalk can be used to overcome that.
  • For Automation Server, other limitations of the API can be overcome using LabTalk.
